Recent development projects

  • (2015-2017) AngularJS / J2EE ERP : As expert and technical architect, I help building a large application for the French public sector. It is an ERP that will handle HR and finance needs. I worked in a team of 5 developers in France and 11 developers in Morocco.
  • (2014) Dashing node.js : I authored six dashboards for development teams and top executives. I had to develop widgets, style adaptations and specific jobs to wire these dashboards to several data sources (Oracle, CI, etc.).
  • (2013) Advanced GWT grid widget : In a team of three persons, I co-authored a grid widget in GWT that let users customize their data with grouping, formulas to change cell appearance, optional columns, grouping functions (average, sum, etc.), etc. Customizations can be saved and recalled, with a credential system to eventually propagate them to other users. This widget is close to the possibilities of a spreadsheet and is used to display management data and operational dashboards.
  • (2007-) Web Architecture : Since 2007, I worked on the creation and maintenance of a Spring/Hibernate/GWT SOA and stateless architecture. This architecture is backed by various hibernate, spring and GWT extensions and propels, as of 2013, 9 business application for more than 2 millions LOC.
  • (2012-2013) UI Unit testing facility : One of the difficulties around automatic UI testing is data management : will a UI scenario create the data it needs as part of the scenario or will it work on predefined data ?
    This testing facility combines Selenium with a meta language based on Excel that let users use template files to easily describe their test case. So, scenario actions (e.g. type, click, hover and so forth) are decorrelated from the data inputs and outpus (e.g. the type action will be run with various data to be typed). The facility then ensures that each test case will work on the data sets defined for the test round.
  • (2011) GWT GUI language : Before UIBinder was released, the need for a higher-level language was already strong in various developments projets around GWT. I authored a GWT rebinding extension letting users design their screens with an XML-based language. Our goals were slightly different than UIBinder : no java needed, non-developer users (i.e. business functionality designers). This language went popular inside the Berger-Levrault group and is now part of the usual design tools for more than fifty persons.
  • (2010-2011) Android agenda : This development was based on the Android 2.x series and aimed to create an agenda with a built-in information system close to the "Agenda-Conseil des Mairies" (English:"Town Hall's Counsellor Agenda") which is a physical agenda. This agenda was a product prototype aimed to city Mayors. It displays reminders and notifications about legal obligations (updated regularly to follow law modifications) for the Mayor and let it, amongst other functionalities, delegate these obligations to his team with a drag and drop operation. Note: France is know for the complexity of its law system, which creates a market for such advice-tools : the first paper edition of this agenda is more than a hundred years old.
  • (2012) Web Architecture : Online public services are growing and Berger-Levrault wanted to propose to public administrations a public website to let administrees pay their bills, see their children's meals and activities, receive collectity news, etc. The architecture to back this project was to combine good UI capabilities and browser compatibility with a very scalable capacity (e.g. Typically, more than 80% of users pay their bills on the same day). I proposed and prototyped a solution around JSF (part of J2EE), a noSQL data source (Amazon Web Services Simple BD) on a no-DAO, bean-oriented and REST model. The project is now in development stage with a five developers team.
  • (2007) Java payroll engine : For payroll engines, performance and customization ease are paramount. This development made use of Javassist to translate payroll rules as classes and therefore generates the specific program to compute a particular payroll before executing it.

The facts behind this résumé

How ?

You'll find here the explanation of how I wanted to code this résumé, as well as some hightlights about its functionalities.

Philosophy

All the source of this résumé has been hand coded : the "how" I wanted to implement is the use of plain HTML and CSS, with no external library, to show that a fair amount of functionalities can be accessed without needing additional scripts. Also, I tried to ensure that semantics and presentation were separated : this provided me the possibility of conceiving different presentations for different format factors and medias. Just try to resize your window when viewing this résumé to see what I mean.

Focus

In this résumé, I included some technical points I wanted to detail for those interested :

  • Flexible layout : Each part of this résumé is not fixed on a predefined grid and flow according to the format factor of the user agent displaying it. Try to zoom or unzoom to see the flow !
  • Responsive design : Different layout are used for different screen sizes. Try extreme resize of this resume to see its differents adaptative designs.
  • HTML5 compliance
  • CSS printing : if you try a print preview of this résumé, you'll see that it is optmized for a standard page size and low ink consumption.
  • Interactivity : Some additional informations can be accessed by hovering elements of this résumé. Also, inside links are featured.

This implementation could be improved, there a some of the ideas that could be coded :

  • Better accessibility : semantics and presentation are well separated but it could go further by using WAI-ARIA recommendations.
  • Rate mechanism : why not increase interactivity by letting user rate, and eventually comment, this résumé. This feedback could be stored in the cloud and retrieved later via web services, as an example.
  • Dynamic theme : I coded this résumé with simple black on white scheme but alternative color schemes could be provided.

In fact, a résumé is a subject that can lead to experiments without end, including many technologies.

Why ?

First of all, the main reason that motivated me in making this web résumé is the fact that, during the five years I was committed to engineer recruitment, all of the résumés I received were more or less the same. This document is meant to display factual data to let the recruiter know better about a candidate. And because a résumé is not associated with an unique formal presentation, this document can be used to demonstrate a certain creativity, details that will make the candidate stand out of the crowd. Thinking about this, I realized that a software developer - specifically with web skills - should develop its résumé.That's why I began writing the code that ended as the résumé you are reading :)
